## Что усвоено (кандидаты → touch-очередь №7)

1. **Leading words + echo-check.** Плотные термины в INPUT ведут агента,
   когда он сам их повторяет в рассуждениях. Наши: «слепая проверка»,
   «конверт», «веер», «линза», «сделка, не инструкция», «дословно».
   Новое: ПРОВЕРЯТЬ эхо — формулирующий называет ведущие термины INPUT'а,
   verify/родитель ищет их отражение в run.log исполнителя. Термин без эха =
   слабое место управления.
2. **Deletion test + осадок (sediment).** Каждый кандидат в конституционный
   touch обязан назвать поведение, которое он меняет («что сломается, если
   удалить?»); правило без поведенческой дельты = no-op, не вносится.
   Каждый touch включает проход на осадок: правила, не процитированные ни
   одним INPUT/VERDICT с прошлого касания — кандидаты на удаление.
   (Родственно принципу «живости» из INFO-010-дискуссии: меняет поведение /
   проверяемо / переживёт масштаб.)
3. **Экономика триггеров.** Модельный вызов = плата контекстом и
   непредсказуемостью (модель может не пойти по указателю); человеческий
   вызов = плата вниманием человека. Принцип binding'а: критичное для метода
   — только конституционное MUST или явный человеческий триггер; никогда
   «модель заметит описание».

## Независимые подтверждения нашей архитектуры (без действий)

- «Прячь будущие шаги — агент выкладывается на текущем» = FEV-разделение
  (VP4: свежий контекст исполнителя и проверяющего).
- «Минимальный SKILL.md, ветки за указателями» = VP20 слим-пакеты +
  закрытые списки чтения (§2 INPUT).
- «Single source of truth» = дисциплина verbatim-записей + побайтовые
  сверки цитат в verify.

## Не берём

User-invoked каталог скиллов как основную модель: наш пользователь —
нетехнический фаундер, его cognitive load — яд; наша ставка — автономия
с обязательными гейтами, не меню.
